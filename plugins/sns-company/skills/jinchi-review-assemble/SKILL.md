# jinchi-review-assemble（工程8 4Kアセンブルの成果物レビュー）

人智の外側 長尺パイプラインの **工程8（4Kアセンブル）** 専用の成果物レビュー。組上げ成果物のゲート。親 `jinchi-longform` の工程8末尾から発動。

## 読むのは3つだけ（混在禁止・token効率）
- ① **本skill内の観点リスト**（下記＝工程8固有の合格ライン）
- ② `.company/pm/review-baseline.md`（全レビュー普遍の7観点・★観点7＝権利・合法性）
- ③ `.company/sns_accounts/youtube_ainsight.md`（人智の外側の戦略・尺前提の照合先）

## フロー（発動＝この順で全実行・fail-closed）
- ① 本skillの観点を成果物（4K書き出し）に1つずつ照合（pass/fail）
- ② `pm/review-baseline.md` の7観点を1つずつ照合（★観点7＝権利・合法性はfail-closed／特に観点4＝youtube_ainsight.md戦略と相違ないか）
- ③ **fail-closed**：1つでもfailなら提出せず→修正→①へ戻る
- ④ 全passのみ次工程（字幕同期）へ進める

- [ ] **★台帳・skillが約束した機構が、この題材のアセンブラに【実在するか】をgrepで確認したか＝系統退行（前弾にあった機構が後継で消える）を疑ったか**：アセンブラは前弾をコピーして作るため、**コピー元が既に機構を落としていると、以降の全題材が黙って機構なしになる**（2026-07-17＝③図解holdは第2弾`build_herculaneum_4k.py`・第3弾`build_ghost_hominin_4k.py`には在るのに`build_valeriana_4k.py`で消え、それをコピー元にした`build_aivuln_4k.py`にも無かった。台帳は「工程8でhold」と約束していたが**実装が存在しなかった**）。**合格＝「台帳/skillが約束した機構名」を新アセンブラに `grep` して実在を確認**（例＝`MIN_FIG_SEC`／`_FIG_REPORT`）。**シンボルが在るだけでは不足＝実際に呼ばれているか（呼出元grep）と、実配分での achieved を実測**まで見る [[reference_jinchi_assembler_hold_and_fit_gates]] [[reference_symbol_presence_is_not_working_code]]

- [ ] **★J: 【台帳の"外"突合③】図解の割付 ↔ アセンブラの実配分での実表示尺を突き合わせたか（工程3から移送・恒久2026-07-22 オーナー確定）**：`jinchi-review-ledger` の「台帳の外と突き合わせているか」観点のうち**③は工程3ではアセンブラ未実装で検証不能**のため、**実配分が初めて存在するこの工程（工程8＝4Kアセンブル）が受け皿**＝ここで実施しなければ誰も検証しない（※オーナー決定文の「工程7＝アセンブル時」は本skillのこの工程＝**親skillの番号では工程8**。工程7＝有料AIカットはHiggsfield解約で消滅）。**合格ライン＝③図解の全カットで「アセンブラの実配分での実表示尺 ↔ 台帳A表の割付尺s」が一致（許容±0.05s）**、かつ**図解のhold退行が0件**（`MIN_FIG_SEC` 割れ・`FIG_HOLD_FLOOR` 割れ）。**測る対象はアセンブラの実出力**＝章別ビルドの dryrun が出す SECTIONS／`_FIG_REPORT`／章JSON `fig_report` の **achieved 実測値**であって、台帳側の設計値同士の照合ではない（それは恒等式＝自作自演）。**台帳の割付と実配分がズレていたら不合格**＝対処は**①アセンブラの配分ロジックを台帳どおりに直す ②台帳の割付を実配分に合わせて再計算する**のいずれかで一致させてから次工程へ進む。プール枯渇で `MIN_FIG_SEC` に届かない図は**取れる最大で止め achieved を実測で残し報告**する（黙って一瞬で消す＝不合格）。台帳§H-1b 外-③／§J-4 の該当行を**この工程で実施済みへ更新**する [[reference_jinchi_assembler_hold_and_fit_gates]]
- [ ] **★I: 最終成果物に含まれる全素材が商用利用可能か（絶対基準・fail-closed）**：①**取得元が `jinchi-longform/references/pipeline.md` 工程5の取得元リストにあるか**（リスト外は不合格）②**ロゴ・企業シンボル・公式シール/紋章が映っていないか**（あれば社名テキスト表示へ置換＝`TEXT_TILES` 方式）③**台帳§Kの全数照合**＝完成物に含まれる素材が漏れなく行を持ち全列埋まっているか（**空欄・「不明」・「—」・行が無い素材は不合格**）。**対象＝本編章の映像/図解/★合成/公式画面 ＋ OP/EDの映像とBGM ＋ 章転換SE ＋ ナレーション音源 ＋ フォント**＝**固定資産（OP/ED）・使い回し・過去に承認済みの素材も免除は無い**。
  - **確認根拠は配布元の一次規約**（URL）＋**素材と同じディレクトリに実在する証跡ファイル**を実パスで確認する。サービス名・ファイル名からの推測、**別サービスの規約の流用は不合格**（実例＝tunetank音源にPixabayの「帰属不要」を当てて著作権違反）。**「確認できない」と「NGと記載されている」は別物**＝第三者取得物は許諾を積極的に確認できることが必要／**正規に提供された実行環境の同梱物**（OS同梱フォント等）は禁止条項が無いことを一次規約で確認できれば可。**確認は選定時点＝事後のまとめ確認は不合格**。
  - **「§D 承認素材 同一性ゲート」のpassは本観点の代替にならない**＝streamhash一致は「オーナーがOKした物と同じ物か」しか示さず権利を検証しない。**両方をpassして初めて結合可**。「オーナーがOKした」「前弾で使った」「再選定しない」は通過理由にならない＝**凍結は制作上の凍結であって権利の再検証は封じない**。
  - **★未確定がゼロか（fail-closed・最重要）**＝「判断が必要」「要オーナー確認」「調査中」「未確認」が**1件でも**残っていれば**総合PASS／GOを出してはならない**。**アップロード（非公開含む）・公開・提示は「公開系の操作」＝未確定が1件でもあれば実行しない。非公開だから良い、は理由にならない**（2026-07-20 オーナー逐語）。取れる道は**①オーナーへ提示して判断を仰ぐ ②その素材を使わない**の2つだけ。
  - **★台帳照合だけでは不合格＝完成物の実フレームを抽出して目視したか**（2026-07-20 オーナー指摘）＝**台帳が「使わない」「テキストのみ」と書いていても実装がそうなっている保証はない**（実例＝台帳が「ファルコンの鷹マークは使わない」と明記していたのに実映像には赤いファルコンが映っていた＝台帳が指定したSVG自体がファルコンを含み、緩和策が最初から実行不可能だった）。**「対策を書いた」と「対策が効いている」は別**。※「画素は作者を証言しない」と**逆方向で両方必要** [[reference_provenance_cannot_be_measured_from_pixels]]。

- [ ] **★L: 絵が、対応するナレーションより先に出ていないか（ネタバレ禁止・恒久）**：**各カットの開始時刻が、そのカットが対応するナレーション文の実発話開始（Whisper onset）以降**であること。**合格＝先行カット0件（許容誤差 -0.05s まで）**。1件でも先行すれば不合格。**★合成・③図解のように答えや核心を見せる絵ほど厳格に**見る（例＝「30年近く続く歴史」の図が、その一文が読まれる3.8秒前から出ている＝不合格）。
  - **★原因の型＝字幕は行頭onsetアンカーなのに、カット（画）の切替が台帳の設計尺による比例配分のままだと、画が音より前に出る**。字幕だけonsetに紐づけて画を比例配分にしている実装は、この観点を構造的に満たせない＝**カット割付そのものを発話onsetへアンカーする**（各カットを台本行へ対応づけ、その行の実発話区間内に収める）。実装の実在を grep で確認する。
  - **検証は章JSONの数値だけで済ませない**＝JSONの `cut_lead_report`（各カットの start / 対応行onset / lead）で0件を確認したうえで、**実フレーム抽出と実音声のonsetを突き合わせ**、核心を見せるカットが対応ナレの発話後に出ていることを現物で確認する。
