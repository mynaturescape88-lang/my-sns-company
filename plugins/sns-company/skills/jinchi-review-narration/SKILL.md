# jinchi-review-narration（工程4 ナレーションの成果物レビュー）

人智の外側 長尺パイプラインの **工程4（ナレーション生成）** 専用の成果物レビュー。音声成果物の品質ゲート。親 `jinchi-longform` の工程4末尾から発動。

## 読むのは3つだけ（混在禁止・token効率）
- ① **本skill内の観点リスト**（下記＝工程4固有の合格ライン）
- ② `.company/pm/review-baseline.md`（全レビュー普遍の7観点・★観点7＝権利・合法性）
- ③ `.company/sns_accounts/youtube_ainsight.md`（人智の外側の戦略・トーン前提の照合先）

## フロー（発動＝この順で全実行・fail-closed）
- ① 本skillの観点を成果物（連結ナレ音声）に1つずつ照合（pass/fail）
- ② `pm/review-baseline.md` の7観点を1つずつ照合（★観点7＝権利・合法性はfail-closed／特に観点4＝youtube_ainsight.md戦略と相違ないか）
- ③ **fail-closed**：1つでもfailなら提出せず→修正→①へ戻る
- ④ 全passのみ次工程（アセンブル）へ進める

## 工程4の観点（合格ラインを内包・1つずつ照合）
- [ ] 全partをloudnorm正規化してから連結したか（音量差なし）
- [ ] 全再生成でなく誤読は文スプライスで直したか
- [ ] **★R: TTS読み精度を【台本×Whisper逐語突合】で検証したか（集計率を合格根拠にしない）**：完成ナレをWhisperで書き起こし、台本(clean)と逐語で突き合わせる。**合格＝台本の語が音声から脱落・置換されていないこと（不一致ゼロ）**。
  - **★読了率・カバレッジ率・文字数比などの集計指標を合格根拠にしてはならない**＝`文字数(whisper)/文字数(台本)` は長さの比でしかなく語の脱落・置換を検出しない（「読了95%」は「5%が誤読/脱落」であって合格ではない／長さが同じなら全く別の内容でも100%になる）。**率が高くても逐語不一致が1件でもあれば不合格**。
  - **判定は【読み（モーラ）】で行い、表記ゆれは不合格にしない**＝Whisperの同音異字・算用数字化はASR側の表記であって誤読ではない（読み一致＝合格の実例＝タンパク質→「単博室/淡白室」・半世紀→「反省器」・アミノ酸→「アミノさん」・五十年→「50年」）。**モーラが欠落/増加していれば誤読**（不合格の実例＝「数ヶ月」→「カ月」＝「すう」脱落）。是正＝TTS入力側の読み固定（REPL等でかなに開く）＋**字幕表示辞書へ復元規則を対で追加**し、再生成後にWhisperで当該語の読みを実測確認する。
  - **助数詞・接頭辞（数ヶ月/数年/数十）・漢数字・単位・専門用語・固有名は1語ずつ個別照合する**＝語ごとに許容変種（同音異字・算用数字）を列挙し、そのいずれも現れなければ不合格。**差分は delete だけでなく replace も見る**（「数ヶ月→カ月」は delete でなく replace として現れるため、脱落検出だけでは取りこぼす）。
  - 走査は**全章**に対して行う（1箇所直して終わりにしない）。結果は章別に「逐語一致率・脱落/置換の実文字列・高リスク語の照合結果」を実測で残す。
- [ ] ピッチ統一せず自然ピッチの良いテイクを使ったか（別人化NG）
- [ ] 表示字幕は元のままTTS入力だけかな化したか
- [ ] 語り部声を口パク内蔵音/保存ボイスで担保したか（Gemini TTSに同一声を期待しない）（新規） [[reference_tts_voice_consistency_and_4k_assembler]]
- [ ] **★スタイル指示文の音声化を【冒頭混入】と【全体置換】の両方で見たか（適用範囲・恒久）**：混入は「冒頭2〜4秒」だけとは限らない＝**テイク全体がstyle文言だけで本文が1文字も読まれていない**破損が起きる（実例＝設計101sの章に対し実尺4.5s・内容は章styleの「この章は〜です」のみ）。**冒頭トリムで直る前提で書かれた下の手順は"冒頭混入"型にのみ有効**＝全体置換型はトリム不可＝**そのテイクを破棄して再生成**する（styleの文言側も見直す＝長いかな括弧などstyleが本文と誤認されやすい要素を外す）。判定＝**章の実尺が設計尺から±50%超**、または**Whisper全文が台本本文と一致しない**なら破損＝不合格。**mp3が存在することを「生成完了」の根拠にしない**（生成器が「最良を採用・要whisper確認」と警告したテイクは必ず実内容を確認する）。
- [ ] **章冒頭2〜4秒に、台本に無い「スタイル指示文」を音声化していないか（既知不具合・恒久チェック）**：Gemini(Charon)がプロンプトに含めたスタイル指示（例「この章はデータの確信です。落ち着いた確信を持って」）をそのまま読み上げてしまうことがある。各章の音声先頭をwhisperで確認し、台本1行目と異なる発話（特に「この章は」で始まる文）が入っていたら不合格→原本を`.orig_preamble_bak`で保全した上で実onset手前(0.2〜0.3s手前)でトリム→トリム直後に再whisperで1行目一致を確認→ステールなwhisperキャッシュ削除→再アセンブル。ここで検出・修正しておけば工程9(caption)まで問題が持ち越されない（2026-07-08/09 ch3・ch5で実際に発生・同型手順で修正）。
- [ ] **「〜分の1」等の分数・比率表現が、同一表記でも登場箇所によって桁読み（例「五分の一」→「いちごぶんのいち」）に化けていないか**：漢字表記の読みは一意でも、Gemini/Charonは文脈（言い直し・強調の短文等）によって同じ表記を桁ごとに読んでしまうことがある＝1回だけ耳/whisperで確認して「正しい」と判定しない。**同一表現が章内に複数回出る場合は全出現を個別に確認する**。誤読を確認したらTTS入力のKANA_OPENへかな開き（例「五分の一」→「ごぶんのいち」）を追記して章を再生成する（字幕表示は従来のアラビア数字表記のまま変えない）（2026-07-09 ch5で実際に発生・修正）。
- [ ] **章冒頭スタイル漏れの修正は必ずpreamble-trim手順（上記）で行い、「クリーンな結果が出るまで生成をやり直す（reroll）」で代替していないか**：rerollは正規手順ではない＝Gemini TTSは同一入力でも毎回話速・間の取り方が変わる確率的生成のため、reroll採用は「読みは直るがテンポが別テイクに化ける」事故を生む（2026-07-09 ghost-hominin ch6で実際に発生＝章尺+24%・正味話速7.36→6.29字/s。原因はreroll採用時にテンポ一致を見ずに最初のクリーンテイクを採用したこと）。
- [ ] **誤読修正等でテイクを差し替えた場合、新テイクの尺（v2整音後）が旧テイクと概ね一致するか（目安±5%程度）を採用前に確認したか**：内容が正しくてもテンポが章単位で有意に変わると単体では気づきにくい劣化になる。乖離が大きい場合は採用せず、preamble-trim等の非破壊手順で旧テイクに近い尺のテイクを直す（2026-07-09 ch6と同型）。

## 原則
- レビューに出す＝既知の改善余地ゼロ・自分で直せる弱点を残さない [[feedback_review_means_finished_no_known_gaps]]
- 要件未達は自分で不合格にして作り直す（NGを「採用に足る」で通さない）[[feedback_strict_self_gate_reject_failures]]
- passした物だけ見せる（failを見せて判断を仰がない）
