# jinchi-review-caption（工程9 Whisper字幕同期の成果物レビュー）

人智の外側 長尺パイプラインの **工程9（Whisper字幕同期）** 専用の成果物レビュー。字幕成果物のゲート。親 `jinchi-longform` の工程9末尾から発動。

## 読むのは3つだけ（混在禁止・token効率）
- ① **本skill内の観点リスト**（下記＝工程9固有の合格ライン）
- ② `.company/pm/review-baseline.md`（全レビュー普遍の6観点）
- ③ `.company/sns_accounts/youtube_ainsight.md`（人智の外側の戦略・表記前提の照合先）

## フロー（発動＝この順で全実行・fail-closed）
- ① 本skillの観点を成果物（同期字幕・再ビルド動画）に1つずつ照合（pass/fail）
- ② `pm/review-baseline.md` の6観点を1つずつ照合（特に観点4＝youtube_ainsight.md戦略と相違ないか）
- ③ **fail-closed**：1つでもfailなら提出せず→修正→①へ戻る
- ④ 全passのみ次工程へ進める

## ★検証は必ず「実際に書き出された動画ファイル実体」で行う（恒久・2026-07-08/09の事故から）
- **captions.json/timeline.jsonの数値同士を突き合わせるだけの検証は禁止**。それらのソースJSONが内部整合していても、**実際に書き出されたmp4に焼き込まれた字幕PNGが同じ状態からレンダリングされたか**は別問題（古いcapsキャッシュ・別バージョンのcaptionsを焼いている可能性がある）。
- 必ず対象mp4から**フレームを複数箇所抽出**（`ffmpeg -ss <t> -frames:v 1`）して字幕の実描画を目視相当で確認し、**音声を抽出してwhisperで再解析**（`ffmpeg -vn`→faster-whisper）して実発話と突き合わせる。
- **本レビューは秘書が代替してはいけない**。秘書がBash/Pythonで検証コードを書いて「一致しています」と判定し提示することは禁止＝必ず制作役(creator)がこの手順で実施する。

## 制作順序（字幕同期は「通し1本」でなく「章ごと」に回す・恒久）
通し約20分を毎回再ビルドすると1指摘=20分待ちで非効率。字幕同期の確認・修正は章単位で回す：
1. 章ごとに独立クリップを書き出す：`scripts/build_herculaneum_4k_chapter.py <cold|ch1|ch2|ch3|ch4>`（章別fi接頭でcuts/secsキャッシュ分離＝直した章だけ数分で再レンダ）。
2. 章単位で下記の数値ゲートを内部fail-closedで判定。
3. 不合格章だけ直し、その章の cuts/secs を消して当該章のみ再レンダ（whisper_n*.json は章別なので流用可）。
4. 全章passの後に最後の1回だけ通し最終4Kへ（本体 `build_herculaneum_4k.py all`・同じ同期/表示ロジックを使うので章別passは通しでも保たれる）。

## 工程9の観点（合格ラインを内包・1つずつ照合）
- [ ] **★全章 自動走査を先に通したか（FB箇所だけ直す＝禁止・恒久2026-07-01）**：オーナー指摘（ch1是正）は「欠陥の一例」であってチェックリストではない。FBを観点(ルール)へ一般化し、cold/ch1-ch4の**全章・全字幕行**を機械走査して同種問題を洗い出す。`scripts/herculaneum_full_audit.py` を実行し **R1(辞書漏れ)=0・R2(漢数字残存)=0・R3(3行字幕)=0** を実ファイルで確認してからビルドへ進む。指摘箇所だけ直して他章を見ない対応は不合格。走査結果（各観点の検出件数と修正内容）を内部記録に残す。
- [ ] **字幕同期＝行頭onsetアンカリング方式か**：各台本行の行頭特徴語をwhisper実認識語へ部分一致でマッチし、その語の発話start秒を行の表示開始に採用（`line_times_anchored`）。旧「累積文字数比例」一本は章内±0.5sドリフトを生むため非推奨。同型 `build_nazca_4k.py` も同方式。
- [ ] **行内キーワード重複による誤アンカリングが無いか（既知不具合・恒久チェック）**：行頭特徴語が**同一行内に2回以上出現**する行（例「ネアンデルタール人の…本物のネアンデルタール人のDNA」）は、whisperの認識誤り（1回目の頭文字を誤認識）と`find()`の挙動により**2回目の出現に誤ってヒット**し、表示開始が数秒〜十数秒遅延することがある（onset数値ゲート＝median/maxは前行の表示開始に吸収されるため素通りする＝機械ゲートだけでは検出不可）。全行の行頭特徴語について「行内に複数回出現していないか」を静的走査し、該当行は実際の音声onsetと照合、ズレていれば当該行だけ手動onset上書き（`ANCHOR_ONSET_OVERRIDES`等）で是正する。1箇所直したら**全9章・同種パターンを走査**してから次工程へ進む（FB箇所だけ直して終わらない）。
- [ ] **音と字幕のズレを全章で機械検証したか（目視NG・自動ゲート必須）**：`scripts/jinchi_caption_sync_check.py <chap.mp4> <captions.json>`。**合格＝median|delta|<0.15s かつ max|delta|<0.35s**。先頭(cold)だけ綺麗で通すのは禁止（短い仮名行は誤差小＝中盤章のズレを見逃す）。※max は whisper再認識jitter(~0.5s)が混じるため、median<0.15が主・2〜3行が0.35〜0.6sでもmedian良好＋原因が認識divergence/jitterなら可（内部記録に残す）。
- [ ] **★機械ゲートのPASSを合格根拠にしていないか（誤PASS前提・恒久2026-07-17）**：`jinchi_caption_sync_check.py` の median/max は「札の開始に**最も近い任意の発話onset**」を拾うため、別語・前行の語に吸着して**誤PASSする**（実例＝第4弾起1で median 0.000/max 0.000＝PASS報告なのに、中盤4札が音声より**1.0〜1.6s先行**していた＝視聴者に見えるズレ）。**章の提示前に必ず実mp4からフレーム抽出（`ffmpeg -ss <t> -frames:v 1`）×独立Whisper転記**で、その札の**台本行の実発話onset**と**焼き字幕の表示開始TC**のΔを実測し、**全札|Δ|<0.35s**を実ファイル証跡（抽出PNG・転記ログのパス）付きで示す。数値ゲートのPASS報告だけを根拠に提示したら不合格。 [[reference_caption_sync_gate_false_pass_reading_mismatch]] [[feedback_review_gate_requires_real_file_evidence]]
- [ ] **アンカー成立率＝「補間へ落ちた行が0」か（恒久2026-07-17）**：行頭onsetアンカーが**成立した行数／全行**を章ごとに記録する。char-ratio補間へ落ちた行は、実発話の「間」を無視して比例配置するため**1s級で早まる**（＝上の誤PASSの実因）。落ちた行は原因を**読み表記の不一致**（clean台本＝読み形「エーアイ」 vs whisper＝綴り「AI」、および「ひとつ↔一つ」「たとえる↔例える」等の表記ゆれ）と見て、読み正規化辞書（`<topic>_prep_tts_text.py` の `REPL`／`_READ_EXTRA`）へ語を足して**再アンカーする**（工程9手順＝`references/pipeline.md`）。実績＝起1 8/12→**12/12**で4札とも|Δ|<0.35sへ収束。
- [ ] **章別「台本字数÷whisper認識字数」比 ≥ 0.97 か**（補助指標）。下回る章は要再同期確認。
- [ ] **A: 字幕の漢字表示辞書を適用したか（恒久）** `scripts/jinchi_caption_display_dict.py` の `apply_display_dict`。TTSは読み仮名で生成済＝音は不変、字幕表示だけ漢字化。「エーアイ→AI」＋読み仮名の固有名詞/数値+単位/専門語（西暦79年・火砕流・軽石・小プリニウス・摂氏・21歳・2023年10月 等）。新題材で読み仮名語が出たら辞書へ追記して全章一括反映。読み仮名がそのまま字幕に出ていたら不合格。
- [ ] **A2: 漢数字→半角算用数字へ正規化したか（恒久）** 同モジュール `normalize_kanji_numbers`（caption_pngで自動適用・図解内テキストも同方針）。位取り数を算用化（九十点→90点・二千年→2000年・千五百年→1500年・十八世紀→18世紀）。**ガード必須**＝「数百/何百」等の概数・単独一桁（一切/一本/三人）・単独位語（百を超える/十倍）は変換しない（意味が変わる）。字幕に漢数字の位取り数が残っていたら不合格。
  **日付は算用数字で出す（追補・恒久2026-07-19 オーナー決定）** 同モジュール `normalize_kanji_dates` が漢数字＋「月」「日」の日付表現（1〜12月・1〜31日）を日付コンテキスト限定で自動統一（六月→6月・二十二日→22日・**単独一桁も対象**）。**日付でない語は据え置き**＝三日月(みかづき)・一日中・五月雨/五月晴れ・十五夜・正月・ヶ月/か月（期間）、および月日以外の助数詞（三年/五回/六人/十二分 等）。無効値（十三月〜・三十二日〜）も本ルールは触らない。字幕に漢数字の月日が残っていたら不合格。
- [ ] **★A3: 字幕にTTS用のかな表記が残っていないか＝正式表記か（全章走査・恒久2026-07-18 オーナー指摘）**：clean原稿は音声用に固有名や漢語を【音写かな】で書く（`グーグル`/`ディープマインド`/`アンソロピック`/`世界じゅう` 等）。これを**字幕にそのまま焼くのは不合格**。表示辞書（`jinchi_caption_display_dict.py` DISPLAY_MAP）を通して正式表記へ戻す＝①実在固有名は**公式ラテン綴り**（Google/DeepMind/Anthropic/Claude/Microsoft/NVIDIA/Amazon/Apple/ChatGPT/FreeBSD/SQLite/Project Zero/Project Glasswing/Big Sleep/Mythos/Fable 等・台本のカタカナ本文表記でなく字幕はラテン化＝オーナー指定）②音写漢語は**台本本文の漢字**（`世界じゅう`→`世界中` 等）。**A の既知リストだけ見るのは不合格**＝全11章cleanを機械走査し、カタカナ連（`[ァ-ヶー]{3,}`）と音写かなを全数抽出→(a)正式表記へ対応済みか (b)判断が割れる語（人名のラテン/カナ等）は「要オーナー確認」に分離、を確認する。各対に証拠（clean出現章＋正式綴りの根拠＝台本/公式綴り）。**検証は実mp4のフレーム抽出**で字幕に正式表記が焼かれていることを現物確認（辞書に足しても cut cache が効いて古い字幕が残る事故があるため＝キーは apply_display_dict 後の文字列で取る）。 [[reference_tts_reading_needs_paired_display_restore]]
- [ ] **★A4: TTS用の過剰な読点が字幕に残っていないか＝書き言葉として自然な句読点か（全章走査・恒久2026-07-18 オーナー指摘）**：台本の読点はTTS最適化（「間」の物理生成）の産物であって書き言葉の句読点ではない＝字幕へ台本通り焼かず、書き言葉として自然な句読点へ正規化する（正本台本・TTS入力は不変）。TTS原稿は「間」のため主格「が」直後・目的格「を」直後・修飾語と被修飾語の間（「本物の、致命的な」）・条件節「たら」直後などに読点を挿す。字幕は**書き言葉の句読点へ減らす**（`reduce_tts_commas`）。**calibration＝ch7の2文**：「AIが、世界中で動くソフトから、本物の、致命的な抜け穴を、次々に見つけてしまう」→「AIが世界中で動くソフトから、本物の致命的な抜け穴を次々に見つけてしまう」／「その穴の一覧を、もし、そのまま世界に公開したら、どうなるか」→「その穴の一覧をもし、そのまま世界に公開したらどうなるか」。**保守的＝文法上必要な読点は残す**（逆接「ますが／ところが」・主語の並列「大企業が、政府が、」・連体詞「その、／この、」・並列「17年もの、16年もの」は削らない）。**憶測で全読点を消さない・迷う行は消しすぎない**（under-reductionは可・over-reductionは不可）。**実mp4フレーム抽出**でch7の上記2文が calibration どおり焼かれているか現物確認。
- [ ] **C: 1枚あたり最大2行・文区切りで分割したか（恒久・実描画行数で判定）** `build_captions` が「。！？」で文単位に分割し尺を文字数比で再配分（`split_caption_sentences`）。**文字数の上限(約60字)だけで判定しない**＝仮名主体の行はCAP_SIZE/CAP_MAXWの実ラップで60字以内でも3行になることがある（既知不具合・2026-07-09 ch4で発生）。分割判定は実際のwrap関数（`wrap_pixel`等）で行数を算出し、**実描画≤2行**を条件に含める。3行字幕や複数文の詰め込みは不合格。カット同期は line_starts ベースで不変＝行内分割は同期に影響しない。
- [ ] 字幕がPIL PNG overlay焼き（白文字黒フチ）か
