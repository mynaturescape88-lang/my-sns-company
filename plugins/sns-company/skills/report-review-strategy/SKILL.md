# report-review-strategy（戦略コメント・見直し判断の成果物レビュー）

SNS運用レポートパイプラインの **戦略コメント・見直し判断（マーケター成果物）** 専用の成果物レビュー。親 `sns-report` のマーケター戦略照合完了後・秘書報告着手前から発動され、戦略判断が根拠と承認ゲートを守るかを合否判定する（Layer2＝工程固有の作り込み合否）。

## 読むのは3つだけ（混在禁止・token効率）
- ① **本skill内の観点リスト**（下記＝戦略コメント固有の合格ライン）
- ② `.company/pm/review-baseline.md`（全レビュー普遍の7観点・★観点7＝権利・合法性）
- ③ 対象プラットフォームの `.company/sns_accounts/<platform>.md`（戦略の唯一の正本・却下レバーの照合先＝`skill-routing.md`で特定。全PF対象時は各該当1枚）

## フロー（発動＝この順で全実行・fail-closed）
- ① 本skillの観点を成果物（戦略コメント・見直し案）に1つずつ照合（pass/fail）
- ② `pm/review-baseline.md` の7観点を1つずつ照合（★観点7＝権利・合法性はfail-closed／特に観点4＝対象platform戦略と相違ないか）
- ③ **fail-closed**：1つでもfailなら提出せず→修正→①へ戻る
- ④ 全passのみ秘書→オーナー報告へ進める

## 戦略照合の整合性チェック観点（最終5項目構成・合格ラインを内包・1つずつ照合・fail-closed）
> 正本＝`.company/secretary/report_format.md`「5項目の厳密定義」（2026-07-16確定）。構成＝①位置付け ②戦略 ③期待値 ④評価 ⑤見直し（旧6項目の④観測は廃止・評価へ統合済）。
- [ ] **5項目構成か**＝各カードが①位置付け／②戦略／③期待値／④評価／⑤見直しの5項目・項番①②③④⑤で、旧構成の残骸（④観測欄・⑤評価/⑥見直しの番号ズレ）が無いか（新規）
- [ ] **①に数値目標があるか**＝狙い一言＋数値目標（KGI¥50,000逆算値／収益PF=目標¥/月＋必要リード指標・送客PF=送客量目標・note=なし）を書き、**その数値目標が台帳 `sns_accounts/<platform>.md` にも永続化**されているか（新規）
- [ ] **②が台帳「今後の戦略」の実施中施策のみと一致**するか＝**完了/失敗/効果測定済の施策（例＝202本運用・掘り起こし・キャプション更新・Gyre）を含まない**か・即興/KGI逆算作文/GO待ち/未実装を混ぜていないか・台帳②とHTML②が一致するか（新規）[[feedback_report_strategy_6fields_and_review_rules]]
- [ ] **原本が現状へ最新化されているか**＝レポート前に各PF実状を確認し、原本の更新漏れ（例＝塗り絵が公開済みなのに原本「未公開」のまま）を放置して②を書いていないか（新規）
- [ ] **③が数値の想定値**か＝その戦略を実施した場合の想定値を**数値で**書き（定性語だけで終えない）、**外部要因項目（投稿本数等）やKGI逆算KPIそのもの（①へ）を③に含めていない**か・記録項目ごとに対応しているか（新規）
- [ ] **④が③との比較で調子評価**か＝③に対し現在地でPFの調子（好調/現状維持/停滞/不調）→理由を述べ、**多期間（前週/先々週/約1ヶ月前）**・**戦略の狙い/期待値を基準（生の増減で判断しない）**・施策ログ照合を織り込み、**独立した観測欄を置いていない**か・KGI未達中に安易な「好調」が無いか（新規）[[feedback_report_eval_no_default_as_expected]]
- [ ] **⑤が今後アクションで確実性重視**か＝端的な箇条書きで、**好調でもKGI¥5万/期待値に未達なら新施策**・現状維持が最善のPF（放置note・主戦場でないThreads）のみ継続・**過去提案全滅の反省＝確実性の高い良案のみ**（薄い数打ち案を並べない）か（新規）
- [ ] **記法統一**か＝増減は▲(増)/△(減)統一・裸の「-」で増減を書かない・前週生値の併記なし・絵文字なし・句点なし・体言止め・収益PFと送客PFで記載方法を変えているか（新規）
- [ ] **カード内の論理整合**＝①目標→②施策→③想定→④評価→⑤手が一本の筋で繋がっているか（①の目標を②の施策で狙い、③がその想定値、④が③との対比、⑤が④を受けた手＝ちぐはぐでないか）（新規）
- [ ] **全カード横断の整合**＝全PFカードで同種の記法・調子の判断基準・多期間の取り方が揃っているか（1枚だけ古い書式/別基準になっていないか）（新規）
- [ ] **憶測混入ゼロ**＝各記載が実データ/原本/施策ログの裏取り済みで、記憶・推測・「なんとなく」を混ぜていないか。**agentの自己申告（週窓修正・原本反映等）はgit diff/実ファイルで現物確認**したか（新規）[[feedback_report_attribution_weekly]]
- [ ] 見直し不要なら原本を変えず監視ログに評価を追記するに留めたか （新規）
- [ ] 見直し要（想定外）なら勝手に原本を変えず「具体修正・推奨1案＋根拠」を作り報告に添えたか（戦略変更は承認ゲート＝賛同後に原本更新）（新規）
- [ ] 提案前に実施済み/凍結を確認したか（該当`sns_accounts/<platform>.md`却下レバー＋`pm/review-baseline.md`観点3／TASKS.md。旧improvement_log.jsonは廃止＝参照しない）[[feedback_check_history_before_proposing]]
- [ ] 却下済みレバーの再提案でないか（手段でなくKGI月¥50,000・収益貢献度で優先順位）[[feedback_youtube_premise_and_rejected_levers]] [[feedback_anchor_proposals_on_true_goal_not_means]]
- [ ] PDCAは各PF戦略ファイル内で完結させ、別台帳（effect_measurement_schedule）を持たない/作らないを守ったか （新規）

## 原則
- レビューに出す＝既知の改善余地ゼロ・自分で直せる弱点を残さない [[feedback_review_means_finished_no_known_gaps]]
- 要件未達は自分で不合格にして作り直す（NGを「採用に足る」で通さない）[[feedback_strict_self_gate_reject_failures]]
- passした物だけ見せる（failを見せて判断を仰がない）
