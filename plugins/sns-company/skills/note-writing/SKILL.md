# note記事 執筆スキル v1.0

note記事（有料の技術体験談／無料の入門記事）を書くときに発動する。
**目的**：creator/CLAUDE.md にルールを書いて毎回読む方式をやめ、note執筆タスクのときだけ本スキルを発動して全ルールを過不足なく適用する。

## 発動時に必ず行う（発動＝適用）

1. 本スキルの全ルール（下記）を読む。
2. 執筆中・執筆後に **「公開前スキャンチェックリスト」「コードサンプルのセキュリティ」** を必ず通す。
3. 提出前に末尾の「自己チェック」で全項目を確認してから PM レビューへ回す。

> このスキルは note 執筆**専用**。Instagram・ショップ記事・アフィリエイト文のルールは含まない（各専用スキルを使う）。

**成果物の置き場**＝`.company/creator/work/note-articles/`（各記事の原稿md・記事内図版png）。

## note記事（有料・¥290／大型記事¥490）
- テーマ：AI活用でブログ・SNS自動化（技術体験談）
- ターゲット：AI活用に興味があるブログ・SNS運営者（植物SNSとは完全分離）
- 構成：無料35〜40%（背景・結果・全体像）＋有料60〜65%（コード・手順・ハマりポイント）
- タイトル：「結果＋数字ベース」「コード公開」を明記。「〜した話」より「〜する方法」が有効
- 価格：基本¥290・大型記事¥490（2026-06-15確定。「広める」優先で低価格帯に統一・¥100は安物感が出るため不採用。正本＝.company/sns_accounts/note.md）
- **ブログ名・SNSアカウント名は記事中に記載しない**：ブログ名（MAINEのShop-List）・Instagramアカウント名（@mynaturescape・@myraisingdiary）はnote記事内で使用禁止。ブログ読者に記事自動生成がバレるリスクを避けるため。「植物ショップ紹介ブログ」「植物Instagram」「子育てInstagram」等の一般的表現に置き換える。コード内の文字列・コメントも同様（`your-blog.example.com`等のプレースホルダーを使用）

## note記事（無料入門記事・月1〜2本）
- 目的：Google流入とAI学習の入口を作る・有料シリーズへの誘導
- テーマ：「始め方」「入門」「手順」系のキーワードを狙う
  例：「Claude APIの始め方」「GitHub Actionsで自動投稿を始める手順」
- 構成：1,000〜1,500文字・ステップ番号付き・末尾に有料記事へのCTA必須
- 価格：無料（有料設定なし）

## note記事 公開前スキャンチェックリスト（必須・全記事共通）

公開前に以下を必ず全文（コードブロック内・コメント・末尾の一言含む）でスキャンする。

```
□ ブログ名（例：MAINEのShop-List）が記載されていないか
□ SNSアカウント名（例：@mynaturescape・@myraisingdiary）が記載されていないか
□ アプリ名・システムユーザー名（例：maine-analytics）が記載されていないか
□ 実ドメイン（例：mynaturescape-event.com）が記載されていないか
□ 実際のID・トークン値（システムユーザーIDなど数値列）が記載されていないか
□ コードコメント・docstring・末尾の一言も対象（本文だけでなく全体をスキャン）
```

発見した場合は汎用プレースホルダー（`your-blog.example.com`・`my-system-user`・`000000000000000000`等）に置き換える。

## note記事コードサンプルのセキュリティルール（必須）

- **APIキー・アクセストークン・パスワードの実値をコードサンプルに記載しない**
- 秘密情報は必ず `os.environ["KEY_NAME"]` / `${{ secrets.KEY_NAME }}` / `"EAAxxxx..."` 等のプレースホルダー形式で示す
- 記事公開前に全コードブロックを目視確認し、実値が混入していないかチェックする

## note記事 ライブラリ・ツール名の記載ルール

ライブラリ名やコマンド名を記載する際は、名前だけでなく**何のためのものか**を括弧書きで補足する。

- ❌ `requests`
- ✅ `requests`（PythonでHTTPリクエストを送るための定番ライブラリ）

初心者読者が「何をインストールするのか」を理解できる形にする。

## note.com投稿時の注意事項

- **コードブロックはnote.comのエディタで手動設定が必要**: Markdownの ` ``` ` はnote.comへの貼り付け時に自動変換されない。バッククォートがそのまま表示されるため、貼り付け後にnoteエディタ上で各コードブロックを手動でコードブロック形式に変換する必要がある
- **引用ブロック（`>`）内のコードブロックは使わない**: `>` 引用ブロック内にコードを含む場合、貼り付けが崩れる。コードを含む補足説明は引用ブロックを使わず、通常の見出し＋本文形式で記載する

## 自動化・技術系note記事のルール（楽天ROOM⑬案件より）

- **公開前に「対象サービスの規約」を必ず確認する**：自動化・スクレイピング・ツール操作を扱う記事は、対象サービスの利用規約を確認し、**規約違反の手順を有料で指南しない**。楽天ROOMへのプログラムによる機械的投稿は規約違反（[[rakuten-room-automation-tos]]）。代わりに「公式API＋手動操作」など規約クリーンな構成に振り替える。
- **無料部分で"手法名・つまずき一覧"まで出さない**：無料で核心（フレームワーク名・解決手順の地図）を出すと、AIで再現されて有料の価値が落ちる。無料は「手作業の大変さ・成果」中心にし、手法・コードは有料へ。
- **主観的なルールは「筆者の場合」と注釈する**：キャプションのNG表現など、人によって変わる方針は「私の場合の方針です／ご自身のスタイルに合わせて」と添え、普遍ルールと誤認させない。
- **記事の核は「記録で裏が取れる一次情報」に置く**：実施記録（task_logs等）で確認できる事実を中心に書く。記憶ベースで断定しない（例「全部手動でやった」等の裏取りできない断定は避ける）。

## note記事の免責・モデル名ルール（楽天ROOM⑬'案件より）

- **定型の法務免責ブロックは付けない**（2026-06-06 オーナー決定で方針変更）：規約違反しないのが前提の記事に「動作保証なし・自己責任」等の重い定型免責を冒頭・末尾に置くと、**かえって不審に見える**。やましいことがない前提なら不要。
  - **ただし安心材料となる事実は本文に自然に溶かす**：「商品取得は公式API・投稿は自分の手で行う＝サイトを機械的に操作する方法ではない」のように、読者の不安（BAN等）を解く事実は本文の一文として残す。これは免責ではなく"売り"。
  - ※この変更は [[feedback_note_article_rules]] の旧ルール「免責を必ず冒頭・末尾に入れる」を上書きする。
- **Claudeのモデル名は最新の正式IDを使う**：オーナーや外部から「古い／別のモデル名では？」と指摘されても、**最新の正式ID（現行は `claude-sonnet-4-6`＝Sonnet 4.6）を優先**する。`claude-3-5-sonnet-20241022` 等は旧世代。読者向けには「実行時点の最新の正式名称に置き換えてください」という注釈コメントをコード内に添えると親切。
- **ベネフィットの数字は実感値ベースでよい**：導入部に時間短縮の目安（例「30件の準備に3〜4時間→貼るだけに」）を入れると有料部分への期待が高まる。盛らず、オーナー提供の実感値・記録で裏が取れる範囲にとどめる。
- **「自分でリンクを渡すだけ」パターンの確実性を推す**：AI検索は意図しない商品ヒットのリスクあり。自分で選んだ商品の紹介文だけAIに書かせる方式は「外しがなく確実・初心者向け」。導入ハードルの低さとして強調する。

## 数字・テーブル・体験記の扱い（MEMORY移送 2026-06-22）

- **数字・事実はコード/実ファイルで確認してから書く**（推測で書かない）。投稿頻度・ファイル名・スケジュール等の数値は実ワークフロー/スクリプトを読んでから記載。「週2本/週3本」のズレは信頼性を損なう。
- **Markdownテーブル・図解はnoteでコピペすると崩れる→Pillow(PIL)でPNG画像化して挿入**。対象＝技術スタック表/費用表/比較表、および全体像・アーキ図（ASCII図でなくPNG）。フォント＝ヒラギノ角ゴW3/W6・背景`#F5F7FA`が雛形。
- **体験記（旅行記等）は"読者が再現できる"を最優先**＝固有名詞＋実体験の因果で書く（抽象化・匿名化しない）。商品/店/ホテル/サービスは実名（公開情報の店名等はマスク不要・個人特定情報のみマスク）。数字は実値＋末尾に「○年○月時点＋免責」。構成＝**冒頭に結論→「正確に記録します」と前置き→スケジュール順（時刻つき時系列）**で漏れなく。日記化はNG（実用ガイドに）。制作後は素材メモと突合し一般化・欠落を監査（[[feedback_experience_article_concrete_reproducible]]）。

## 提出前 自己チェック（PMレビュー前に全項目）

```
□ 有料/無料の構成比・価格・CTAは規定どおりか
□ 公開前スキャン6項目を全文（コード・コメント・末尾含む）で通したか
□ コードに実APIキー/トークン/パスワードが無いか（プレースホルダー化したか）
□ ライブラリ名に用途の括弧書きを付けたか
□ note.com貼り付け前提（コードブロック手動変換・引用内コード禁止）を考慮したか
□ 規約違反の手順を有料指南していないか
□ モデル名は最新正式ID（claude-sonnet-4-6）か
□ 重い定型免責を付けていないか（安心材料は本文に自然に溶かす）
```
> 「レビューに出す＝作り込み済み・既知の改善余地ゼロ」（[[feedback_review_means_finished_no_known_gaps]]）。

## ◆成果物／・作業ノードの分類
- ◆構成・タイトル・価格設定（→ `note-review-structure`）
- ◆本文（→ `note-review-body`）
- ◆公開前スキャン・コードセキュリティ（→ `note-review-scan`・★秘匿の最終ゲート）
- ・note.com投稿操作（純作業＝レビューskillを当てない／コードブロック手動変換・引用内コード禁止は本文/scanで担保）

## 子レビューskill 発動連鎖（成果物ノードごと・fail-closedで次へ）
- 構成・タイトル・価格確定後 → `note-review-structure` を発動（pass のみ本文へ）
- 本文執筆後 → `note-review-body` を発動
- 本文完成後・公開操作前 → `note-review-scan` を発動（秘匿/秘密の最終ゲート）
- 全◆pass後・公開直前 → `pm/review-baseline.md`（Layer1てっぺん総合）を1回当てる
※公開前の法令/著作/事実は content-compliance。投稿操作は作業ノード＝レビューskillを当てない
