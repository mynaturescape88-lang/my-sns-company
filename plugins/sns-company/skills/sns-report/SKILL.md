# SNS運用レポート作成スキル v2.0（オンデマンド・決定的）

SNS運用レポートの作成・数値分析を依頼されたときに発動する。
**目的**：レポート手順を「誰がいつ実行しても同じ結果（決定的）かつ高速」にする。前回表示したHTMLをベースに、数値を最新化し戦略照合の差分だけ更新して表示する。毎回ゼロ生成しない・過去分は再集計しない・多段の往復をしない。

## 大原則（オーナー確定 2026-07-15）

- **オンデマンド**：レポートは依頼された時にその場で作る（自動集計前提は廃止）。
- **フォーマットは固定＝変更一切NG**：HTMLの構造・レイアウト・列/行・カード構成・項目セットを変えない。最新化するのは**セルの値**と**戦略照合の指定テキスト**だけ。
- **前回HTMLがベース**：`seo_growth/latest_report.txt` が指す前回表示HTMLを土台に最新化する。数値は差分だけ動かす＝速い。
- **決定性の担保**：入力（前回HTML）→処理（該当セルのみ更新→合計再計算→戦略照合の指定部のみ更新）→出力（固定フォーマット）を一意に定義。数値最新化は**スクリプト一発**（下記B）。
- **取得不可セルは埋めない**：未集計・API取得不可のセルは無理に埋めず現状維持（"-"）でよい。
- **性能契約＝依頼→Chrome表示まで≦60秒**（目標）。**1分以内に出せない場合はプロセスが誤り**（多段往復・逐次取得・ゼロ再生成が混入している）。取得が遅いPF（特に楽天Playwright）は**タイムアウトしてスキップし、そのセルは現状維持でHTMLを出す**（「取得できないセルは埋めない」を適用＝1PFの遅延で全体を止めない）。

## 決定的フロー（A→B→C→D→E・この順で固定）

### A. 入力を一意に確定（前回HTML＝ベース）
- `seo_growth/latest_report.txt` を読む＝ベースにする前回表示HTMLの絶対パスを一意に特定（無ければ `seo_growth/sns_monthly_report_*.html` の最新mtime）。
- そのHTMLを開き、どのPF・どの項目が埋まっているか（空白セル／既存値）を確認する。

### B. 数値を一発で最新化（作業ノード・往復なし・並列取得）
- **これ1コマンドで決定的に完了**（単一オーケストレーション）：
  ```
  .venv/bin/python seo_growth/render_report_preserving_strategy.py --no-open
  ```
  interpreter は**リポジトリ直下の `.venv`**（google-analytics-data 等入り）。`seo_growth/.venv` はyt-dlp用でGA4が動かないので使わない。
- このスクリプトが決定的に行うこと：
  1. 全SNSの実数値をライブ取得（`collect()`＝youtube/ig/ga4/adsense/search_console/rakuten/note の各report）。**各PF取得は互いに独立＝並列に走らせ、各取得に上限タイムアウトを付す**。**楽天Playwright（最重・約30秒）がタイムアウトしたらスキップ**しそのセルは現状維持でHTMLを出す（1PFの遅延で全体を止めない）。
  2. 当週セルを `report_history.json` に記録（`record()`）＋取得可能な過去週の空セルのみ遡及充填（`backfill()`）。**過去分の再集計はしない**（`calendarize` は呼ばない＝既知の巻き戻しバグを回避）。**フロー数値の週窓＝暦週(月〜日)の直近完了週＝`wk_done=monday_of(today)-7`**（`record()`と一致・ローリング7日/API遅延バッファ禁止＝当週混入や週合算二重計上を招く）[[reference_report_flow_window_must_match_wk_done]]。
  3. 数値表・合計/小計/週合計/MTD を hist から再計算（決定論）。
  4. 各月の**戦略照合ブロックは前回HTMLからverbatim抽出して温存**（splice）＝フォーマットも戦略テキストも1文字も変えずに数値だけ最新化。
  5. 出力HTMLの絶対パスを標準出力に印字（`integrated_out_path`＝同一月構成なら同一パスに上書き）。
- **月次コスト（固定費）・収支（KGI対比）は触らない**：コード内固定値。オーナーからの申告があった時のみ `monthly_report_build.py` の `MONTHLY_COSTS` を修正する（毎回自動で触らない）。
- 検証だけしたい時（APIを叩かず現状 hist から温存レンダ）：`--render-only --no-open`。

> ✅ **並列取得は実装済み（2026-07-15）**：`monthly_report_build.collect()` は各PF取得を `ThreadPoolExecutor` で並列起動し、各取得にタイムアウト（既定50s／楽天Playwrightは40s）を付す。超過取得は `"__ERR__"` を返し `parse_*` が全項目None→`record()` がskip＝該当セルは現状維持（1PFの遅延で全体を止めない）。`youtube_report.py` の2回叩き重複も解消済（1回の結果を本体MAINE／人智の外側で共有）。**実測 54.8s→14.6s ＝ ≦60秒契約を安定達成**（読み取りAPIのみ・0cr・冪等・描画バイト不変）。

> Bの出力時点で「数値は最新・戦略照合は前回のまま」。フォーマットは固定。次のCで戦略照合の指定部だけ差分更新する。

### C. 戦略照合を差分だけ最新化（マーケター・簡潔）
Bで温存された戦略照合ブロックのうち、**指定項目だけ**を最新化してHTMLに直接書き込む。**STEP0.5で原本を事実へ最新化済みの前提**で、各PFの `sns_accounts/<platform>.md`（原本「今後の戦略」）を読み、記憶や即興でなく原本から引く。項目ごとの扱い：

| 項目 | 扱い |
|---|---|
| ①位置付け | **狙い一言＋数値目標（KGI¥50,000逆算値をここに置く）**。収益PF=目標¥/月＋必要リード指標の数値／送客PF=送客量目標／note=数値目標なし。KGI逆算配分（正）＝MAINE¥23,000／人智¥10,000／楽天¥9,000／Kids¥5,000／AdSense¥3,000（note除外）。数値目標は台帳にも永続化。原則不変・流用 |
| ②戦略 | **`sns_accounts/<platform>.md`「今後の戦略」の実施中施策のみ**（即興・KGI作文NG／完了・失敗・効果測定済は施策ログへ・②に載せない＝例202本運用/掘り起こし/キャプション更新/Gyre／GO待ち・未実装も載せない）。**原本が現状と食い違うなら先に原本を最新化してから引く**（更新漏れNG＝STEP0.5）。台帳②とHTML②一致。変更なければ不変 |
| ③期待値 | **その戦略を実施した場合の想定値＝数値**（記録項目ごと）。**定性語（「維持」「伸ばす」の語のみ）禁止＝必ず数値**。外部要因項目（投稿本数等）は置かない。★**KGI逆算のKPI（月◯再生・GMV¥◯等）は①に置き③へ混入させない**。施策を増やしたらその分を追記 |
| ④評価 | ③に対し現在地でPFの**調子（好調/現状維持/停滞/不調）**→理由。**多期間（前週/先々週/約1ヶ月前）・戦略の狙い/期待値基準（生の増減で判断しない）**・**施策ログの実施日×現在数値**の帰属切り分けを織り込む（測定未到来なら再判定日）。**独立した観測欄は置かない（旧④観測を廃止＝評価内で③と対比）**。数値はHTMLの表セルと一致必須。**KGI未達中に安易な「好調」を付けない** |
| ⑤見直し | 評価を踏まえた今後のアクションを端的な箇条書き。**好調でもKGI¥5万/期待値に未達なら新施策**（「好調＝現状継続」を安易にデフォルトしない）。現状維持が最善のPF（放置note・主戦場でないThreads）のみ継続。**過去提案全滅の反省＝確実性の高い良案のみ**。定型文（「見直し不要・監視のみ」等）・作業メモ・却下済み施策・却下レバー列は書かない |

- 記法は固定フォーマットに従う（簡潔な箇条書き・複数項目や①②③列挙は `<br>`・句点「。」不要＝体言止め/短句・増減は▲(増)/△(減)統一で裸の「-」禁止・前週生値の併記なし・絵文字なし・収益PFと送客PFで記載方法を変える・カード項番①②③④⑤）。
- 戦略**原本**（`sns_accounts/<platform>.md`）は勝手に書き換えない＝⑥で新施策を提案する場合はオーナー賛同後に原本更新（戦略変更は承認ゲート）。ここではHTML上の戦略照合セルを更新するだけ。

### D. レビュー（fail-closed・全pass以外は表示しない）
成果物ノードごとに正規レビューskillを1巡当てる。全passでなければ修正してから再度当てる。
1. 数値分析（アナリスト成果物）→ `report-review-analysis`（数値整合・実行順序）
2. 戦略照合（マーケター成果物）→ `report-review-strategy`（戦略照合・承認ゲート）
3. 全◆pass後・表示前 → `pm/review-baseline.md`（Layer1てっぺん総合）を1回
- **提出前ゲート**：全記載を実API/スクリプトの実値・`sns_accounts`原本・各台帳で裏取り（古いスナップショット/静的memoに依存しない）。②戦略は実施済みのみ。全項目の裏が取れるまで「完了」と言わない。

### E. 表示＋保存（今回結果が次回ベース）
- Chrome表示：`open -a "Google Chrome" "<Bで印字された出力HTMLの絶対パス>"`
- **保存＝次回ベースの更新**：`seo_growth/latest_report.txt` を今回の出力HTMLの絶対パスに更新する。これで今回の（数値最新＋戦略照合更新済み）HTMLが次回のベースになる（次回Bのsplice温存元）。
- 秘書はHTMLを表示するだけ＝チャットで数値の長文要約・表の再掲はしない [[feedback_report_show_html_not_chat_summary]]。

## 性能契約（依頼→Chrome表示 ≦60秒・超過はプロセスの誤り）
律速を作らないための構造：
- 数値最新化は**Bのスクリプト一発**（並列取得＋タイムアウト付き）で決定的に完了。役割エージェントの多段往復で数値を再集計しない。
- 戦略照合は**Cで差分だけの単一パス**（①位置付け・②戦略・③期待値は原則不変で流用＝④評価・⑤見直しのみ更新）。マーケターの往復ループを回さない。
- レビュー（D）は**軽量1巡**＝各レビューskillの観点を1回ずつ当てるだけ。原本の重い再読込を繰り返さない（STEP0で読んだプロファイルを使い回す）。
- フォーマット固定・前回HTMLベースなので「バグ発覚→ゼロからやり直し」が起きない。

**超過時の切り分け（≦60秒を超えたら、まずどこで詰まったかを特定）：**
1. **B（取得）が遅い**＝最有力。`collect()` は並列化済みなので、通常は楽天Playwrightがタイムアウト上限まで粘っているケース。→ タイムアウトが効いているか確認し、必要なら楽天だけスキップして他PFで60秒に収まるか確認（楽天セルは現状維持でよい）。
2. **C（戦略照合）で往復している**＝マーケターが原本を何度も読み直す/推敲ループ。→ 差分項目（④⑤）だけに絞り単一パスで書く。
3. **D（レビュー）で重い再取得**＝レビューが数値を再集計している。→ レビューは整合チェックのみ（再集計しない）。
4. **A/Eの取り違え**＝ベースHTMLをlatest_report.txtから取らずゼロ生成している/保存先を誤っている。→ A・Eの一意パスを再確認。

## STEP0（Cの前提・スキップ厳禁）
- 戦略照合（C）の前に、対象PFの `.company/sns_accounts/`（README＋関係アカウント）を読む。各アカウントの位置づけ・沿革・過去施策・戦略を取り違えると照合が失敗する（特にYouTube＝「植物ショップ長尺で収益化した本体／子供ショートは後発の派生」を読み違えない）。

## STEP0.5（①②の前提・スキップ厳禁＝台帳を「実施中施策＋数値目標」へ最新化してから①②を書く）
- ②戦略は**原本 `sns_accounts/<platform>.md`「今後の戦略」の実施中施策のみ**を参照して書く（記憶・即興・KGI逆算の作文は禁止／完了・失敗・効果測定済は施策ログへ・②に載せない）。参照前に**各PFの実状を確認し、原本が現状と食い違えば先に原本を最新化する**（例＝塗り絵が既に公開済みなのに原本が「未公開」のまま等の**更新漏れを放置して②を書かない**）。原本の書き換えが事実追認でなく**戦略変更**に当たる場合はオーナー賛同後に更新（承認ゲート）。
- ①の**数値目標（KGI¥50,000逆算配分＝MAINE¥23,000／人智¥10,000／楽天¥9,000／Kids¥5,000／AdSense¥3,000・note除外・送客PFは送客量）は台帳 `sns_accounts/<platform>.md` にも永続化する**（今まで抜けていた＝①に書くだけでなく原本にも残す）。
- 不明点だけ `TASKS.md`／`TASKS_COMPLETED.md`／`task_logs/_INDEX.md` から該当sessionlogを**1件だけ**引く（全ログ通読しない）。
